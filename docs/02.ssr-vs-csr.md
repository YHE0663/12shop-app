# CH01-07. SSR vs CSR

## 🎯 프로젝트 요구사항

1. 서비스가 더 많은 사용자에게 **노출**될 수 있도록 한다.
2. 사용자에게 더 많은 **서비스 정보를 제공**한다.
3. 사용자에게 서비스의 **매력을 증진**시킨다.

→ 핵심 키워드: **“더 많은”, “노출”, “매력”**  
→ 이를 기술적으로 연결하는 첫 단계는 **SEO(Search Engine Optimization)**

---

## 🔍 SEO (Search Engine Optimization)

> 웹사이트가 검색엔진에 **잘 노출되도록** 만드는 기술적 활동 전반

### 주요 구성 요소

| 요소                             | 설명                                                                            |
| -------------------------------- | ------------------------------------------------------------------------------- |
| **Cross Linking**                | 특정 페이지가 다른 여러 페이지로부터 링크될수록 중요도가 높게 평가됨            |
| **Title Tag & Meta Description** | 각 페이지의 제목과 메타 정보를 명확히 기술해 검색엔진 점수 향상                 |
| **URL Canonicalization**         | 동일한 콘텐츠가 여러 URL로 노출될 때, 대표 URL로 통합 관리하여 검색 정확도 개선 |

### 추가 평가 요소

| 항목                           | 설명                                                         |
| ------------------------------ | ------------------------------------------------------------ |
| **HTTP Version (Secure Site)** | HTTPS를 통한 안전한 통신은 신뢰성과 품질 향상                |
| **Page Speed**                 | 페이지가 빠르게 로딩될수록 사용자 경험과 검색 랭킹이 개선됨  |
| **Structured Data**            | HTML 문서 내 데이터 구조가 명확하고 시멘틱하게 구성되어야 함 |
| **Mobile Compatibility**       | 모바일 친화적 사이트일수록 검색엔진에서 높은 점수 부여       |

> ✅ 요약: SEO는 **안전성(HTTPS) + 속도(PageSpeed) + 구조화(Structured Data) + 모바일 대응(Mobile)**  
> 이 네 가지 축을 중심으로 페이지 품질을 평가한다.

---

## 💡 Showcase WebApp 개요

> **목표:**  
> 12shop.com 모바일 앱의 **쇼케이스 지면의 웹 버전**을 개발하여  
> 웹을 통한 사용자 유입 확대와 서비스 소개 역할을 강화한다.

- 간단한 서비스 소개와 앱 다운로드 링크 제공
- 웹 방문자에게 더 많은 서비스 정보를 제공 → **노출 및 매력 향상**
- SEO 대응이 가능한 SSR 방식으로 검색엔진 접근성 확보

---

## ⚖️ CSR vs SSR 비교

### 🔹 CSR (Client Side Rendering)

| 항목     | 내용                                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------------------------- |
| **장점** | - Backendless (서버 의존도 낮음)<br>- CDN, Cache 전략 활용 가능<br>- 초기 로딩 이후 빠른 인터랙션                   |
| **설명** | 모든 애플리케이션 코드가 JS 번들 파일 안에 포함되어, CDN 배포만으로도 동작 가능.<br>트래픽, 서버 부하에서 자유로움. |
| **단점** | - 초기 로딩 최적화에 많은 비용<br>- SEO 대응 한계<br>- 멀티 도메인 API 연동 시 추가 아키텍처 필요(CORS, Gateway)    |
| **요약** | 빠르지만, 검색엔진 노출이 제한적이고 구조가 복잡해질 수 있음.                                                       |

---

### 🔹 SSR (Server Side Rendering)

| 항목     | 내용                                                                                                                                 |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **장점** | - SEO 대응 용이<br>- 빠른 초기 로딩 (조건적)<br>- 멀티 도메인 API 연동 용이                                                          |
| **설명** | 서버에서 HTML을 완성해 전달하므로, 브라우저·크롤러 모두 완전한 문서를 받을 수 있음.<br>API 호출도 서버 간 통신으로 보안·구성이 단순. |
| **단점** | - 렌더링 백엔드 필요<br>- 복잡한 아키텍처 구성<br>- 서버 트래픽 부담                                                                 |
| **요약** | 완전한 SEO 대응이 가능하지만, 운영·구성이 복잡하고 비용이 증가함.                                                                    |

---

## 🧭 Showcase의 기술 선택

> **현재 상황:**  
> 서비스는 초기 단계이며, 사용자 수는 많지 않지만  
> **검색 노출과 유입 확대**가 중요하다.

→ 따라서 **SEO가 강점인 SSR이 적합**

> “검색엔진에 더 잘 노출되고, 더 많은 사용자가 쉽게 접근할 수 있도록 한다.”

---

## ⚙️ SSR 구현 방법 (React 기반)

| 구현 방식                          | 설명                                                                             | 특징                                       |
| ---------------------------------- | -------------------------------------------------------------------------------- | ------------------------------------------ |
| **React 직접 구현 (Node.js 기반)** | React가 제공하는 저수준 API(`renderToString`)를 활용하여 직접 HTML을 생성        | 학습용으로 적합 (SSR 내부 구조 이해 가능)  |
| **Next.js 활용**                   | React 기반의 SSR 프레임워크로 라우팅, 빌드, SEO, Image Optimization 등 통합 제공 | 실전용으로 적합 (빠른 시작, 유지보수 용이) |

> ✅ **결론:**
>
> - **React SSR → 학습용** (아키텍처 구조 이해에 도움)
> - **Next.js → 실전용** (서비스 운영 효율성 높음)
>
> 현재 Showcase WebApp은 **학습 목적**의 SSR 구조를 직접 구현하며  
> 이후 **Next.js 전환을 고려**하는 방향이 이상적이다.

---

## 🧾 정리

- **요구사항 → SEO로 연결**
- **CSR ↔ SSR 비교를 통해 SSR 채택**
- **React SSR 구현으로 구조 이해**
- **Next.js는 실전 배포용으로 고려**

> 💬 **한 줄 요약:**  
> “12shop Showcase는 SEO 친화적 SSR 구조를 통해  
> 더 많은 사용자에게 노출되고, 서비스의 매력을 높이는 웹앱을 목표로 한다.”

# CH04-03. 웹뷰 웹앱 설계 및 구성 (CSR 선택)

## ⚖️ CSR vs SSR – 웹뷰 관점에서의 판단

웹뷰 내 웹앱은 일반적인 웹 서비스와 전제가 다르다.  
이번 프로젝트 기준으로 **CSR(Client Side Rendering)** 방식을 선택한다.

---

## ✅ CSR 선택 이유

### CSR #1

### 독립된 웹뷰 수 증가 예상

- 웹뷰 화면은 하나가 아닐 수 있다.
- 독립된 웹뷰가 여러 개로 스케일 아웃될 가능성이 있다.
- SSR 선택 시:
  - 각 웹뷰마다 **렌더링 서버 의존**
  - 백엔드 운영 비용 증가
  - 백엔드 장애 시 서비스 안정성 저하

→ 웹뷰 수가 늘어날수록 **SSR 운영 부담이 급격히 증가**

---

### CSR #2

### SEO 필요성이 낮음

- 웹뷰 내 웹앱은 **검색 엔진 노출 대상이 아님**
- 앱 내부 특정 화면이 검색 엔진에 노출되는 것은 오히려 바람직하지 않다.
- SEO가 중요한 일반 웹과는 정반대의 상황

→ 웹뷰 환경에서는 **SEO 대응을 고려할 필요가 낮다**

---

### CSR #3

### 기민한 사용자 반응성이 중요

- 웹뷰 내 웹앱은:
  - 단일 화면일 수도 있고
  - 탭 인터페이스를 가진 다중 화면일 수도 있다.
- 한 번 로딩 이후:
  - 화면 전환
  - 탭 이동
  - 사용자 인터랙션

→ **CSR이 SSR보다 반응성이 뛰어남**

> 웹뷰에서는  
> 초기 렌더링보다 **즉각적인 반응성**이 더 중요하다.

---

## 📌 결론

- 이번 웹뷰 프로젝트는 **CSR 아키텍처 채택**
- 모든 웹뷰에 통용되는 선택은 아니다.
- **현재 프로젝트 특성에 맞춘 결정**이다.

---

## 🏗️ 프로젝트 구성

### 📱 React Native (App)

1. **Expo**
2. **React Native UI Library**
3. **Expo WebView**

#### Expo 선택 이유

- React Native 프로젝트 환경을 빠르게 구성 가능
- CLI 기반 보일러플레이트 제공
- 복잡한 환경 설정 없이 개발 시작 가능
- CRA와 유사한 역할의 도구

---

### 🌐 WebView 내부 WebApp

1. **Create React App (CRA)**
2. **Webpack**
3. **React App Rewired**
4. **Storybook**
5. **Chakra UI**

---

## 🔧 Webpack 언급 이유

- CRA는 내부적으로 Webpack을 사용하지만 설정을 숨긴다.
- 기본적으로 Webpack을 직접 다룰 필요는 없다.
- 하지만 **번들러 선택은 프로젝트 규모가 커질수록 중요한 의사결정 포인트**가 된다.

### 관점

- 현재는 CRA + Webpack 구성
- 향후 프로젝트에서는:
  - 번들러 선택지를 한 번쯤 고민해볼 필요가 있다.

> 지금은 설정을 숨긴 CRA를 사용하지만  
> 번들링 전략 자체는 **언젠가 반드시 고민해야 할 영역**이다.

---

## 🧾 정리

- 웹뷰 환경에서는 CSR이 합리적인 선택일 수 있다.
- SEO보다 **운영 비용, 안정성, 반응성**이 중요하다.
- App은 React Native + Expo
- WebView는 CRA 기반 CSR 웹앱
- 기술 선택은 항상 **프로젝트 맥락 기준**으로 이루어져야 한다.

> 기술은 정답이 아니라  
> **상황에 맞는 선택의 문제**다.
