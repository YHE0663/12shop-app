# CH01. PoC vs Prototype

## 🎯 개념 정리

### 🔹 PoC (Proof of Concept)

> 아이디어가 **기술적으로 실현 가능한가**를 확인하는 단계

- **목표:** 프로젝트의 불확실성을 제거하고, “이 기술로 가능한가?”를 검증
- **핵심:** 구현 가능한 기술과 **구현 가능성이 미지수인 기술을 분리**
- **주요 질문 예시:**
  - 이 기술은 실제 서비스에 쓸 수 있을 만큼 안정적인가?
  - 성능이 요구 수준에 도달할 수 있는가?
- **예시:**
  - 이미지 서버 개발 시 → “N개의 리사이징 이미지를 빠르게 생성할 수 있을까?”
  - “이미지 최적화 옵션을 적용했을 때 성능이 충분한가?”  
    → 이런 문제를 코드로 **짧게 실험**해서 판단하는 것

> 💬 **핵심 키워드:** “기술 검증”, “불확실성 제거”, “실험 코드”

---

### 🔹 Prototype

> PoC 이후, 아이디어의 **정합성과 가치를 빠르게 검증**하는 단계

- **목표:** 아이디어를 **실체화**하여 “이 방향이 맞는가?”를 확인
- **차이점:** 기술적 가능성(PoC) → **사용자/컨셉 검증**
- **중요한 건 기술이 아니라 시간**
  - 아이디어의 가치를 가장 빠르게 확인할 수 있는 방법을 선택
  - “완성도”보다 “속도”가 우선
- **예시:**
  - 네이티브 앱으로 만들 예정이라도,  
    컨셉 확인을 위해 **웹앱으로 빠르게 구현** 가능
  - 사용자 입장에서는 네이티브/웹 구분보다  
    “이 서비스 컨셉이 괜찮은가?”가 더 중요

> 💬 **핵심 키워드:** “아이디어 검증”, “컨셉 실체화”, “속도 중심”

---

## 🧩 PoC vs Prototype 요약

| 구분      | PoC                  | Prototype                 |
| --------- | -------------------- | ------------------------- |
| 목적      | 기술적 가능성 검증   | 아이디어/컨셉 검증        |
| 초점      | 불확실성 제거        | 사용자 가치 확인          |
| 코드 성격 | 실험용, 버려도 됨    | 시제품, 발전 가능         |
| 기준      | “될까?”              | “이게 맞을까?”            |
| 속도      | 중요하지만 기술 중심 | **가장 중요 (빠른 검증)** |

---

## 💻 그래서 지금 showcase 웹앱은?

- showcase는 “**Prototype**” 단계입니다.  
  이미 사용할 기술(Vue, React, Tailwind, Nest 등)은 명확하므로  
  이제 **서비스 아이디어의 정합성**을 빠르게 확인하는 게 목적이에요.

### ✅ 지금 해야 할 일

1. **핵심 시나리오 1개만 빠르게 완성하기**
   - 예: “상품 리스트 → 상세 페이지”
   - 동작만 가능하면 OK (디자인/성능은 나중)
2. **네이티브 수준의 UX를 의식하지 않아도 됨**
   - 앱처럼 보여도 **웹으로 빠르게 만들어 확인**하면 충분
3. **사용자 반응을 통해 방향 검증**
   - 클릭 흐름, 데이터 흐름만 확인하면 됨

> 💬 **정리하자면:**  
> 지금 showcase 웹앱은 기술 검증(PoC)이 아니라 **“아이디어의 방향을 눈으로 확인하는 Prototype”**이에요.  
> 목표는 “이 컨셉 괜찮다!”를 빠르게 증명하는 것 ⚡️
