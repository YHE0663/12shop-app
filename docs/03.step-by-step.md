# CH01-08. Step 1 – 개발

## ❓ 개발할 때 중요한 것은 무엇인가?

### 흔히 나오는 답변

- 어플리케이션 아키텍처
- 시스템 아키텍처
- 프로그래밍 테크닉 (SOLID 등)
- 일정, 안정성, 확장성, 유연성

> 물론 중요하지만, **실제 실무에서 더 본질적인 요소**는 따로 있다.

---

## 💡 강의에서 말하는 핵심 관점

### 1️⃣ 자산 인지 기반의 활동 능력

> 개발은 항상 **제한된 자산** 위에서 이루어진다.

- 시간
- 사람
- 기술력
- 운영 비용

→ 현재 내가 가진 자산을 **정확히 인지하고**,  
그 범위 안에서 **가장 합리적인 선택을 할 수 있는 능력**이 중요하다.

---

### 2️⃣ 이슈에 대한 선제적 커뮤니케이션

> 이건 역량이라기보다 **태도에 가까움**

- 문제가 생겼을 때 혼자 끌어안고 오래 고민하지 말 것
- **시간을 정해두고**, 해결되지 않으면 바로 공유
  - “이 부분이 막혔는데, 어떻게 접근하는 게 좋을까요?”
- 혼자 오래 해결해도 팀 전체 관점에서는 의미가 없을 수 있음
- 일정, 플래닝을 무너뜨리는 가장 큰 원인이 됨

> 💬 **중요 포인트:**  
> 문제를 드러내는 것을 부끄러워하지 말고, 빠르게 공유하라.

---

### 3️⃣ 개발자가 챙길 수 있는 것 챙기기

> 아래 항목들은 **개발자가 아니면 챙기기 어렵다**

#### 1. 테스트

- 단위 테스트(Unit Test)
- E2E 테스트
- 테스트에 대한 명확한 관점이 있어야 어플리케이션이 단단해진다.
- “나중에 하자”가 아니라 **처음부터 하나씩 꼼꼼히**

#### 2. 로그

- 사용자 로그
- 애플리케이션 로그
- 장애, 이슈 발생 시 원인을 추적할 수 있어야 함

#### 3. 예외 상황 처리

- 404, 500 등 기본적인 예외 처리
- 에러가 났을 때 “아무 일도 없는 것처럼” 보이지 않게

#### 4. 코드에 매몰되지 않기

- 코드는 수단이지 목적이 아니다.
- **잘 만들어진 제품 + 잘 운영되는 서비스**가 핵심
- 운영이 되어야 개발도 지속될 수 있다.

---

### 4️⃣ 지속적인 진행 상황 가시화

> “한 달 반 뒤에 한 번에 보여주는 방식”은 위험하다.

- 누구나 언제든지 진행 상황을 확인할 수 있어야 함
- 기능 단위, 단계 단위로 계속 공유
- 제품의 진행 상황, 일정, 리스크를 **항상 보이게 유지**

> 💬  
> “짜잔 다 됐어요” 방식은  
> → 막판에 수많은 문제를 한 번에 터뜨릴 가능성이 크다.

---

## 🧾 정리

- 개발은 기술 이전에 **자산과 상황을 인지하는 능력**
- 문제를 숨기지 않고 **빠르게 공유하는 태도**
- 개발자가 아니면 챙기기 어려운 것들(테스트, 로그, 예외 처리)을 놓치지 않기
- 코드보다 **제품과 운영**을 우선시
- 진행 상황은 항상 **가시화**

> 💬 **한 줄 요약:**  
> “좋은 개발은 잘 짠 코드가 아니라,  
> 팀과 제품을 망치지 않는 선택의 연속이다.”

# CH01-09. 어떻게 테스트할 것인가?

## 🎯 테스트에 대한 기본 관점

- 테스트는 필요하다.
- 하지만 **모든 것을 테스트하려고 하면 유지 비용이 더 커진다.**
- 따라서 **프론트엔드 특성에 맞는 실용적인 조합**이 중요하다.

---

## 🧪 테스트 종류 정리

### 1️⃣ 단위 테스트 (Unit Test)

> **핵심 비즈니스 로직 중심**

- 모든 코드에 단위 테스트를 작성할 필요는 없다.
- **핵심 로직**, 비즈니스 규칙이 담긴 부분에만 작성
- 유지되었을 때 명확한 베네핏이 있는 테스트만 남긴다.

✅ 예시

- 계산 로직
- 조건 분기
- 상태 변화 로직

📌 중요 포인트

- 단위 테스트는 **배포 파이프라인에 포함**
- 테스트 실패 시 배포 차단 → 안정성 확보

✍️ 테스트 작성 방식

1️⃣ 스크립트 기반
• 코드로 직접 테스트 시나리오 작성
• 가장 일반적이며 유지보수가 용이

2️⃣ 레코딩 기반
• Cypress에서 제공하는 UI 기반 테스트 레코딩
• 빠르게 테스트 코드 작성 가능

⸻

💡 개발자가 테스트를 작성해야 하는 이유
• QA가 수동 테스트를 수행하더라도 한계가 있다.
• 개발자가 E2E 테스트의 상당 부분을 작성해 두면:
• 반복 테스트 감소
• 시간, 인력, 자원 절약
• QA는 예외 케이스에 집중 가능
• 경험적으로, 특별한 케이스를 제외하면
E2E 테스트만으로도 많은 영역이 커버된다.

⸻

🧾 정리
• 프론트엔드 테스트의 중심은 E2E 테스트
• 단위 테스트는 핵심 로직에만 선택적으로
• API는 실제 호출이 아닌 Mock
• 테스트는 품질 확보뿐 아니라 자산을 절약하는 수단

테스트는
모든 것을 검증하는 도구가 아니라,
사용자 관점에서 앱이 깨지지 않음을 보장하는 장치다.

---

### 2️⃣ E2E 테스트 (End To End Test)

> **프론트엔드의 기본 테스트 축**

- 사용자에게 실제로 **렌더링되는 결과**를 기준으로 테스트
- 화면이 정상적으로 뜨는지
- 주요 기능이 실제 사용자 흐름대로 동작하는지
- 시각적으로 문제가 없는지

❗️주의

- 여기서 말하는 E2E 테스트는 **전체 시스템 통합 테스트가 아니다**
- 실제 API 호출 ❌
- **API는 모킹(Mock) 처리**

👉 즉,

> “프론트엔드의 맨 앞단에서 앱이 정상적으로 동작하는가?”를 확인하는 테스트

---

## ⚖️ 단위 테스트 vs E2E 테스트 정리

| 구분            | 단위 테스트 | E2E 테스트       |
| --------------- | ----------- | ---------------- |
| 대상            | 핵심 로직   | 사용자 관점 기능 |
| 범위            | 좁음        | 넓음             |
| 목적            | 로직 안정성 | 앱 동작 보장     |
| API             | 직접 호출   | Mock             |
| 프론트엔드 비중 | 보조        | **핵심**         |

> ✅ **결론:**
>
> - 핵심 로직 → 단위 테스트
> - 기본 품질 보장 → E2E 테스트 중심

---

## 🛠️ Cypress 기반 E2E 테스트

현재 프로젝트는 **Cypress** 기반으로 E2E 테스트가 세팅되어 있다.

### 실행 방법

```bash
npm run test
```
